## 0. 한 줄 정의

**HawkEye-VLM** = 카메라로 본 드론을 보고, **(1) 적/아군/민간** 식별 → **(2) 위협등급** 판정 → **(3) 회수/투하 결심**을 자연어 근거와 함께 출력하는 온디바이스 VLM.

---

## 1. 왜 VLM이냐 (Why a VLM, not just a detector)

그냥 객체탐지(YOLO 등)로 "드론이다!" 까지는 쉽다. 근데 우리가 알아야 하는 건 더 깊다:

- 이게 **정찰**인지 **자폭**인지 → 밑에 달린 게 카메라 짐벌인지 폭발물 페이로드인지 **봐야** 안다.
- **아군/민간 택배 드론**인지 → 로고, 형상, 비행패턴을 **이해**해야 안다.
- "왜 그렇게 판단했는지" **근거(설명가능성)** 가 있어야 군이 신뢰한다.

> 즉, **분류(detection)** 를 넘어 **이해(reasoning)** 가 필요하다. 그래서 VLM.

## 2. 모델 아키텍처 (Architecture)

```
   [입력]                  [HawkEye-VLM]                       [출력]
 ┌─────────┐    ┌──────────────────────────────────┐   ┌────────────────┐
 │ EO 카메라│──▶│ Vision Encoder (ViT)              │   │ class: 적 정찰  │
 │ IR 카메라│──▶│   + 멀티스펙트럼 융합            │──▶│ threat: LOW    │
 │ 거리/속도│──▶│ Cross-Attn ↔ LLM Backbone        │   │ action: 회수    │
 │ (센서토큰)│   │ (경량 디코더, 인스트럭션 튜닝)   │   │ reason: "짐벌만│
 └─────────┘    └──────────────────────────────────┘   │  있고 폭발물X" │
                                                         └────────────────┘
```

| 구성 | 사양/선택지 |
|------|-------------|
| Vision Encoder | ViT 계열 (SigLIP/EVA류), EO+IR 듀얼 스트림 융합 |
| 언어 백본 | 경량 LLM (2~7B, 양자화) — 온디바이스 추론 가능 |
| 센서 토크나이저 | 거리/속도/RF메타데이터를 토큰으로 주입 (멀티모달) |
| 출력 헤드 | 구조화 JSON (class/threat/action/reason/confidence) |
| 추론 엔진 | TensorRT-LLM, INT8/FP8 양자화, Jetson Orin 타겟 |

## 3. 출력 스키마 (Structured Output)

모델은 **항상 기계가 파싱 가능한 JSON**으로 결심을 뱉는다:

```json
{
  "target_id": "trk_00472",
  "object_class": "enemy_fpv_kamikaze",
  "iff": "hostile",
  "threat_level": "HIGH",
  "payload_assessment": "원통형 페이로드 + 배선 식별, 기폭장치 의심",
  "disposition": "CAPTURE_AND_DROP",
  "drop_reason": "그물째 회수 시 모선 근접 기폭 위험. 무인지대 투하 권장.",
  "confidence": 0.87,
  "requires_human_approval": false
}
```

### 3.1 `object_class` (식별 클래스 예시)
- `enemy_recon` 적 정찰 드론
- `enemy_fpv_kamikaze` 적 자폭 드론
- `enemy_relay` 적 통신중계 드론
- `enemy_swarm_unit` 적 군집 자식기
- `friendly` 아군 드론 (IFF 일치)
- `civilian_delivery` 민간 택배/촬영 드론 (포획 금지!)
- `bird` 새 (제발 잡지 마)
- `unknown` 미상 (보수적 처리)

### 3.2 `disposition` (처리 결심)
| 값 | 의미 |
|----|------|
| `CAPTURE_AND_RETURN` | 그물 포획 후 **기지 회수** (양품, 분석가치 ↑) |
| `CAPTURE_AND_DROP` | 그물 포획 후 **무인지대 투하** ("가져오면 ㅈ됨") |
| `IGNORE` | 아군/민간/새 → 건드리지 않음 |
| `HOLD_FOR_HUMAN` | 애매함 → 인간 최종승인 대기 (HITL) |

## 4. "ㅈ됨 판정기" 로직 (Threat Disposition Classifier)

회수할지 버릴지 결정하는 핵심 의사결정 트리:

```
포획 대상 확정
   │
   ├─ 폭발물/기폭장치 징후 있음? ──── 예 ──▶ CAPTURE_AND_DROP (버린다)
   │        (원통 페이로드, 노출배선,
   │         과도한 무게중심, 열징후 등)
   │
   ├─ 아니오
   │    │
   │    ├─ 분석가치 있음(신형/정보장비)? ─ 예 ─▶ CAPTURE_AND_RETURN (가져온다)
   │    │
   │    └─ 그냥 흔한 양산 드론? ──────────────▶ CAPTURE_AND_RETURN
   │
   └─ 확신 부족 (confidence < 임계) ──────────▶ HOLD_FOR_HUMAN (사람 불러)
```

> 모토: **"의심스러우면 가져오지 말고 버려라. 모선 밑에 매달려 터지는 것보단 들판에서 터지는 게 낫다."**

## 5. 학습 데이터 (Training Data)

| 데이터 종류 | 내용 |
|-------------|------|
| 드론 형상 DB | 적/아군/민간 드론 다각도 EO+IR 이미지 |
| 페이로드 라벨링 | 카메라짐벌 vs 폭발물 페이로드 구분 라벨 |
| 합성 데이터 | 시뮬레이터(에어심/블렌더)로 희귀 자폭드론 증강 |
| 하드 네거티브 | 새, 풍선, 비닐봉지, 택배드론 (오인 방지용) |
| 비행 패턴 시계열 | 정찰(맴돔) vs 자폭(직선돌진) 궤적 |

> 특히 **"새를 적 드론으로 오인"** 하는 사고를 막기 위해 비둘기/까치/독수리 데이터 ㅈㄴ 많이 넣음. (독수리한테 그물 던지면 동물보호단체가 우리한테 그물 던짐)

## 6. 온디바이스 제약 (On-Device Constraints)

| 항목 | 목표 |
|------|------|
| 추론 지연 | ≤ 100 ms / frame (Jetson Orin) |
| 처리율 | ≥ 15 FPS 추적 유지 |
| 모델 크기 | 양자화 후 ≤ 8 GB |
| 오프라인 동작 | **클라우드 없이 100% 온디바이스** (재밍 환경) |
| OTA 업데이트 | 신형 적 드론 등장 시 모델 무선 업데이트 |

---

## 7. 신뢰성 & 사람 개입 (Trust & HITL)

- **설명가능성:** 모든 결심에 `reason` 자연어 근거 첨부 → 군 지휘관 신뢰 확보.
- **HITL 옵션:** 교전규칙(ROE)에 따라 "포획 전 인간 승인" 강제 모드 지원.
- **블랙박스:** 모든 판단 영상+JSON 로깅 → 사후 검증/책임소재 명확.
- **오인 페일세이프:** `civilian_delivery`/`friendly`/`bird`는 **하드락(절대 포획 금지)**.

---

## 8. 평가 지표 (Eval Metrics)

| 지표 | 목표 |
|------|------|
| 적/아군/민간 식별 정확도 | ≥ 98% |
| 자폭 페이로드 탐지 재현율(recall) | ≥ 95% (놓치면 ㅈ됨) |
| 민간/새 오인율 (false capture) | ≤ 1% |
| 회수/투하 결심 정확도 | ≥ 97% |
| 추론 지연 | ≤ 100 ms |

> ⚠️ 본 문서는 **기획 단계 모델 설계**입니다. 실제 모델 학습/벤치마크는 Phase 2에서 진행됩니다.
