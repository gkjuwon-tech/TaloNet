# 🛡️ 05. 방어 시스템 — 항법/링크 보안 (Defensive Stack)

> 우리는 **공격은 안 한다.** 적 드론을 해킹/스푸핑/재밍하지 않는다.
> 대신 **우리 자신**(Peregrine 모선드론)이 스푸핑·재밍·명령 위조에 안 당하도록 **방어 코드로 떡칠**한다.
> 이 문서는 `defense/` 패키지의 실제 구현과 그 근거(검증된 공개 프로젝트)를 정리한다.

---

## 0. 원칙: "내 뇌피셜 금지, 검증된 것만"

방어 코드는 직접 발명하지 않았다. 전부 **공개·검증된 알고리즘/스펙/오픈소스**에 기반한다:

- **Galileo OSNMA** — 항법메시지 암호 인증 (`galileo-osnma` Rust / `OSNMAlib` Python)
- **MAVLink 2 Message Signing** — 공식 스펙(mavlink.io), PX4/ArduPilot 채택
- **RAIM** — 수신기 자율 무결성 감시 (Parkinson & Axelrad, RTCA DO-229)
- **ArduPilot/PX4 GPS glitch & EKF innovation gating** — 양산 비행제어 펌웨어의 실제 보호로직
- **IPsec anti-replay window** (RFC 4303/6479), **HMAC** (RFC 2104)

각 모듈의 출처/라이선스는 [`defense/README.md`](../defense/README.md) 표 참조.

---

## 1. 위협 모델 (Threat Model)

| 위협 | 설명 | 방어 레이어 |
|------|------|-------------|
| **GPS 스푸핑** | 가짜 위성신호로 우리 위치를 속여 추락/이탈 유도 | OSNMA 인증 + RAIM + 일관성 검사 |
| **GPS 재밍** | 신호 마비 | C/N0 붕괴 탐지 → 관성항법 전환 |
| **명령 위조/주입** | 가짜 MAVLink 명령으로 모선 탈취 시도 | MAVLink2 서명 인증 |
| **리플레이 공격** | 정상 명령 패킷 재전송 | 단조증가 타임스탬프 + 슬라이딩 윈도우 |
| **링크 오버파워/탈취** | 더 센 송신기로 링크 가로채기 | RSSI 스텝 + CRC 에러율 감시 |

> 핵심 통찰: **완벽하게 자기일관적인 스푸핑**은 RAIM·일관성 검사로 못 잡는다. 그걸 잡는 유일한 레이어가 **암호 인증(OSNMA)** 이다. 그래서 다층 방어(defence-in-depth).

## 2. 레이어별 구현 매핑

### 2.1 OSNMA / TESLA 항법메시지 인증 — `defense/gnss/osnma_adapter.py`
- TESLA 단방향 키체인(`K_i = H(K_{i+1})`) + 지연공개 MAC 검증의 **핵심 프리미티브**를 의존성 없이 구현(테스트용·오프라인용).
- 실제 위성신호(SIS) 인증은 검증된 **OSNMAlib**(`osnma` 패키지)에 위임 — `osnmalib_available()`로 감지.
- 효과: 적이 자기일관적 가짜 별자리를 만들어도, **암호 서명이 우리 신뢰앵커(K_0)로 체인되지 않으면 거부.**

### 2.2 RAIM (무결성 감시 + 고장위성 배제) — `defense/gnss/raim.py`
- 가중최소제곱 해 → 잔차제곱합(SSE) → **카이제곱 검정**으로 고장 탐지.
- 탐지 시 단일 위성 배제(FDE) 후 재계산. (카이제곱 CDF/임계값은 불완전감마함수로 자체 구현)
- 한계: 단일고장 보호용. 그래서 OSNMA·일관성과 **조합**해서 씀.

### 2.3 스푸핑/재밍 일관성 검사 — `defense/gnss/spoof_detection.py`
양산 펌웨어가 실제로 쓰는 독립 검사들을 묶음:
- **Glitch gate** (ArduPilot): 이전 위치+속도로 예측한 지점과 새 fix 비교, `GPSGLITCH_RADIUS`(5m)+가속도 한계 초과 시 거부.
- **EKF innovation gate**: IMU 예측 대비 정규화 혁신제곱(NIS)이 게이트 초과 시 거부.
- **C/N0 이상탐지**: 비정상적으로 강하고 균일한 신호 = 스푸핑 의심 / 전반적 붕괴 = 재밍.
- **클럭 점프 탐지**: 발진기로 설명 안 되는 수신기 클럭 바이어스 급변 = 탈취 징후.
- **다중위성군 교차검증**: GPS/Galileo/GLONASS 독립 fix 불일치 시 부분 스푸핑 의심.

### 2.4 C2 링크 인증 (MAVLink 2 서명) — `defense/link/mavlink_signing.py`
- 공식 스펙 그대로: `signature = sha256_48(secret + packet + link_id + timestamp)`, 13바이트 서명블록(linkID 1 + timestamp 6 + signature 6).
- 타임스탬프는 2015-01-01 기준 10µs 단위 48비트, **스트림별 단조증가** → 리플레이 거부.
- 효과: **32바이트 공유키로 서명 안 된 명령은 전부 거부.** 가짜 `COMMAND_LONG`으로 모선 못 뺏음.

### 2.5 RF 링크 보안 — `defense/link/rf_link_security.py`
- `AuthenticatedChannel`: HMAC-SHA256 프레임 인증 + 시퀀스.
- `ReplayWindow`: IPsec식 슬라이딩 윈도우 안티리플레이(재정렬 허용, 중복/만료 거부).
- `LinkIntegrityMonitor`: CRC 에러버스트(재밍) + RSSI 스텝(탈취) 통계 탐지.

### 2.6 통합 판정 — `defense/monitor.py`
모든 레이어를 융합해 **설명가능한 항법신뢰 상태**를 반환:

| 모드 | 의미 |
|------|------|
| `TRUST_GNSS` | 정상 |
| `GNSS_DEGRADED` | 주의(공분산 팽창) — 예: 일관성 의심이나 OSNMA 인증됨 |
| `DEADRECKON` | GNSS 무시, IMU 관성항법으로 코스팅 |
| `RETURN_TO_HOME` | 항법+명령링크 동시 손상 → 최후 페일세이프 |

> 설계 원칙: **의심되면 GNSS를 믿지 말고 관성항법으로.** 조용히 가짜 fix를 신뢰하는 일은 절대 없음.

## 3. CONOPS 연계: 작전 중 스푸핑 당하면?

```
05:42  정상 순찰 (TRUST_GNSS)
06:10  적이 GPS 스푸핑 시도
        → C/N0 비정상 균일 + 클럭점프 + glitch 동시 플래그
        → OSNMA 인증 실패(가짜 신호는 서명 못 함)
        → DefenseMonitor: DEADRECKON 전환, IMU로 코스팅
06:11  적이 가짜 명령 주입 시도
        → MAVLink 서명 불일치 → 명령 거부
06:12  관성항법으로 안전구역 이탈/귀환 (임무 지속)
```

적은 우리를 **속이지도(스푸핑 거부), 뺏지도(명령 거부)** 못한다. Peregrine는 계속 일한다.
