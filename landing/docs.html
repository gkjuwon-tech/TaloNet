Technical documentation

How TaloNet actually works

A counter-UAS mothership that nets a hostile drone in flight and recovers it intact — no hacking, no jamming, no spoofing. This page documents what is in the repository, written from the source.

Repository status: design stage. The defensive stack (defense/), the manual cockpit (gcs/), the exploitation pipeline (forensics/) and the parametric CAD (cad/) are implemented and tested. Flight/payload firmware and the PCB/gerbers are future work. Specifications below are design targets.

1 — Concept & ethics

Existing counter-UAS answers either destroy the drone (debris, lost intelligence, poor cost ratio) or attack the radio spectrum (jamming/spoofing — legally fraught and disruptive to friendly comms). TaloNet takes a different posture: a larger mothership drone patrols, an operator identifies a hostile UAS, and a software-aimed net launcher physically captures it. The catch is then either recovered to base intact (reusable airframe + exploitable intelligence) or, if judged dangerous (e.g. a one-way attack drone), dropped safely in an unpopulated area.

The ethical line is enforced in the architecture, not just the marketing:

  • We never transmit against the target. Nothing in the repo jams, spoofs or talks to a hostile drone. Capture is purely mechanical.
  • defense/ hardens us, not them. It only detects and rejects attacks against our own interceptor (GNSS spoofing/jamming, command-link injection/replay).
  • forensics/ reads what we caught. It exploits the storage of an already-captured drone to produce target intelligence — never a fire order. Engagement stays with a human commander under the rules of engagement and the Law of Armed Conflict.

4 — defense/ — navigation & link security

Defence only: nothing here transmits, jams or spoofs. Each module implements a published, verified algorithm or delegates to an audited open-source implementation, and the detection/auth core uses only the Python standard library so it can run on an embedded flight computer without a BLAS stack.

ModuleWhat it doesBased on
gnss/osnma_adapter.pyGalileo OSNMA navigation-message authentication (TESLA one-way key chain + delayed MAC); delegates live signal-in-space auth to a real OSNMA library when installedgalileo-osnma, OSNMAlib; Galileo OSNMA ICD
gnss/raim.pyReceiver Autonomous Integrity Monitoring: least-squares-residual fault detection + exclusion with a chi-square testParkinson & Axelrad; RTCA DO-229
gnss/spoof_detection.pySensor-consistency checks: position-glitch gate, EKF innovation gate, C/N0 anomaly, clock-jump, multi-constellation cross-checkArduPilot GPS glitch/EKF gating
link/mavlink_signing.pyMAVLink 2 message signing (SHA-256, 48-bit signature, monotonic 48-bit timestamp) for C2 auth + replay rejectionMAVLink message-signing spec
link/rf_link_security.pyTransport-agnostic HMAC frame auth, IPsec-style sliding-window anti-replay, jamming/takeover monitorRFC 2104 HMAC, RFC 4303/6479
monitor.pyFuses every layer into one explainable navigation-trust verdictdefence-in-depth (this repo)

The fused verdict (DefenseMonitor.assess)

Every epoch the flight stack calls assess() with the latest OSNMA, RAIM, spoof and command-link results and receives a NavSecurityState whose mode drives behaviour. The logic is fail-safe toward inertial — it never silently trusts a suspect fix:

ModeMeaningTriggered when
TRUST_GNSSUse the GNSS fix normallyno spoof flag, RAIM healthy, link OK
GNSS_DEGRADEDUse with caution / inflate covariancespoof suspected but OSNMA-authenticated, or OSNMA present yet unauthenticated
DEADRECKONCoast on IMU, ignore GNSS positionspoof suspected without auth, RAIM unresolved fault, or jamming
RETURN_TO_HOMELast-resort failsafecommand link lost while already dead-reckoning

Layering is intentional: cryptographic OSNMA authentication is the only layer that can catch a perfectly self-consistent spoof, while RAIM and the consistency checks catch faults and spoofs before OSNMA lock. Link signing is orthogonal and protects the command path regardless of GNSS state. Full OSNMA signal-in-space verification is the one thing deferred to an external audited library (osnmalib_available()); the rest runs dependency-free.

5 — gcs/ — manual teleop cockpit

A ground-control app you install on a laptop. A realistic EO/IR FPV feed fills the screen under a military HUD (boresight + pitch ladder, bank arc, heading/speed/altitude tapes, a HOSTILE-UAS box, and the green NET-AIM reticle); you fly the mothership and aim/fire the net from the keyboard.

Controls

KeysAction
W/S A/Dpitch / roll
Q/Eyaw
R/Fthrottle up / down
I/K J/Lnet aim tilt / pan (slews the reticle)
SPACEfire net (armed only, in the aimed direction)
C / Vcinch (purse shut) / release (drop)
G · B · Harm toggle · E-STOP · return-to-home
N · ESCreset · quit

Interlocks mirror the hardware design: FIRE / CINCH / RELEASE require ARMED, and nothing arms while E-STOP is latched.

6 — forensics/ — counter-UAS exploitation

Exploitation, not intrusion. Nothing here boots, talks to, jams or spoofs a device. It exploits the storage/logs of a drone that has already been non-kinetically captured to produce target intelligence — chiefly the hostile launch site with an uncertainty radius — for a proportionate, human-authorized response.

The seven-stage pipeline

ForensicPipeline owns ordering and integrity; each stage is a Protocol wrapping a verified OSS tool, and the pipeline never writes to the original evidence.

intake → imaging(write-blocker) → hash(SHA-256) → content
      → flight-log → trajectory → report
StageWhat happens
1 — intakeRecord receipt in the chain-of-custody log
2 — imagingAcquire a write-blocked read-only image of the original
3 — hash_verifyIntegrity gate: original & image SHA-256 must match, else IntegrityError and the run stops — no report
4 — contentFilesystem/metadata analysis + log discovery on the image
5 — flight_logParse each recovered log with the matching verified parser
6 — trajectoryReconstruct the track, estimate launch/target, render a map
7 — reportAssemble the threat-intel report + custody appendix

Modules

FileRole
interfaces.pyData contracts + stage Protocols (Imager, HashVerifier, ContentAnalyzer, FlightLogParser, TrajectoryAnalyzer, ReportBuilder)
geo.pyDependency-free trajectory math: haversine, bearing, bounding box, loiter/dwell detection
chain_of_custody.pyChainOfCustody — append-only, hash-chained, tamper-evident custody log

Heavy libraries are imported lazily, so import forensics works with nothing installed; the parser/report/map adapters light up once requirements-forensics.txt is installed. Estimates (launch / target / intent) always carry a confidence and an evidence basis — never asserted as certainty — and the launch point is meant to be cross-checked by independent means.

ForensIQ-1 appliance

A sealed field kiosk (modelled in cad/forensic_appliance.scad): insert the captured drone's microSD/SD into the write-block slot and ForensicAppliance.process_card() runs the whole pipeline, prints a threat-intel report on the built-in 80 mm thermal printer, and archives the PDF to an evidence USB.

7 — cad/ — parametric models

OpenSCAD parametric models for massing, mounting geometry and packaging validation (not production stress models). Both pass OpenSCAD's manifold check.

Key airframe parameters are fully customizable — e.g. wheelbase 1150 mm, prop_dia 711 mm (~28″ folding), capture-net mouth radius/depth 430/560 mm, and net_pan/net_tilt aim angles that map directly onto the gcs/payload_map.py servo limits above. Changing a parameter reconfigures the model (e.g. a larger wheelbase or a non-coaxial hexa).

8 — Testing & CI

CI (.github/workflows/ci.yml) runs on every push to main and every pull request: it lints defense forensics gcs tests with ruff, installs the verified-OSS forensics parsers, and runs the unittest suite on Python 3.12.

# lint + full suite, exactly as CI runs it
ruff check defense forensics gcs tests
pip install -r requirements-forensics.txt
python -m unittest discover -s tests -v

The suite lives in tests/ (12 modules) and covers the defensive layers (OSNMA adapter, RAIM, spoof detection, MAVLink signing, RF link security, the fused monitor), the cockpit control core and MAVLink link, and the forensics core, geo math and parsers. Because the cores are dependency-free, most of the suite runs without any third-party packages installed.
